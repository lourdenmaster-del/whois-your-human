# Narrative / Voice Styling Pipeline — Audit

**Purpose:** Map the end-to-end path that produces user-facing text for Beauty profiles, and identify the smallest leverage point to improve voice quality.

---

## 1. Pipeline Step Map

```
[1] User submits birth data
        ↓
[2] POST /api/beauty/create → POST /api/engine
        ↓
[3] POST /api/engine/generate (engine report)
        │   • ENGINE_SPEC (system)
        │   • User prompt: birth data + "Generate full report + emotional_snippet"
        │   • Output: full_report, emotional_snippet (JSON)
        │   • Second LLM call: IMAGE_PROMPT_SPEC → image_prompts
        │   • Third LLM call: VECTOR_ZERO_SPEC → vector_zero (three_voice + beauty_baseline)
        │   • saveReportAndConfirm → Blob (ligs-reports/{reportId}.json)
        ↓
[4] Engine route fetches report from GET /api/report/{reportId}
        ↓
[5] E.V.E. filter (OpenAI)
        │   • System: EVE_FILTER_SPEC + "STRICT OUTPUT: only { image, report }"
        │   • User: full_report + emotional_snippet + vector_zero
        │   • Output: { image: string, report: string } — forced by route, NOT full BeautyProfile
        ↓
[6] Engine route builds BeautyProfileV1
        │   • filterOutput = minimal stub (light_signature.raw_signal = parsed.report; rest empty)
        │   • buildBeautyProfile(filterOutput, vectorZeroFromReport)
        │   • payload: emotionalSnippet from report; fullReport = parsed.report; vector_zero from engine
        │   • saveBeautyProfileV1 → Blob (ligs-beauty/{reportId}.json)
        ↓
[7] User views /beauty/view?reportId=… → GET /api/beauty/{reportId} → BeautyViewClient renders
```

## 4. Current Prompts & Templates

### 4.1 Engine Report (full_report + emotional_snippet)

**File:** `app/api/engine/generate/route.ts`  
**System:** `lib/engine-spec.ts` — `ENGINE_SPEC`  

**User prompt (inline, line 115):**
```
Generate the full Light Identity Report and emotional snippet for this birth data using LIGS Engine v1.1 (canonical 14-section structure + Cosmology Marbling Patch):

${birthData}

Output valid JSON only with exactly these keys: "full_report" (string, the complete 14-section report: Initiation, Spectral Origin, Temporal Encoding, Gravitational Patterning, Directional Field, Archetype Revelation, Archetype Micro-Profiles, Behavioral Expression, Relational Field, Environmental Resonance, Cosmology Overlay, Identity Field Equation, Legacy Trajectory, Integration. In EVERY section: RAW SIGNAL with 1 subtle cosmological echo; CUSTODIAN with 1 ancient physiological mirror; ORACLE with full fusion of physics, metaphysics, and human meaning—cosmology woven into sentences, not listed. Tone: mythic-scientific, elegant, readable. The full_report must be a long string (multiple paragraphs; no shortening).) and "emotional_snippet" (string, 1-2 declarative sentences). No other text.
```

**Relevant ENGINE_SPEC excerpts:**
- 14-section structure, 3-voice (RAW SIGNAL, CUSTODIAN, ORACLE)
- Physics layer mandatory
- Archetypes: canonical 12 (no injection of per-archetype voice)
- Tone: "Scientific-mythic", "Declarative", "Precise", "No metaphors", "No spirituality", "No personality-test language", "No predictions", "No narrative storytelling", "No filler"
- Cosmology Marbling Patch: RAW = physics + 1 echo; CUSTODIAN = biology + 1 mirror; ORACLE = full fusion

**User prompt:**

Derive Vector Zero from this Light Identity Report. Output only the JSON object.

### 4.3 E.V.E. Filter

**File:** `app/api/engine/route.ts` lines 149–163  
**Spec:** `lib/eve-spec.ts` — `EVE_FILTER_SPEC`

**System (route override, line 154):**
```
${EVE_FILTER_SPEC}

STRICT OUTPUT: You must respond with valid JSON only. No prose, no markdown, no comments, no trailing commas, no additional fields. The only allowed shape is: {"image":"<base64 or URL>","report":"<string>"}
```

**User prompt:**
```
Transform this LIGS output into the Beauty-Only Profile. Output only a single JSON object with exactly two keys: "image" (base64 or URL string) and "report" (string). Nothing else.

Full report:
${fullReport.slice(0, 8000)}

Emotional snippet: ${emotionalSnippet}

${vectorZeroFromReport ? `Vector Zero (use this for vector_zero section):\n${JSON.stringify(vectorZeroFromReport)}` : ""}
```

**EVE_FILTER_SPEC (lib/eve-spec.ts)** describes full extraction (light_signature, archetype, deviations, corrective_vector, imagery_prompts) but the route's STRICT OUTPUT forces `{ image, report }` only. The spec is effectively ignored for the output shape.

### 4.4 Archetype Voice Anchors

**Used by:** `POST /api/voice/generate`, marketing overlay, LIGS Studio  
**Not used by:** Engine report, E.V.E., Beauty profile text  

`getArchetypeAnchor(archetype)` → `{ emotional_temperature, rhythm, lexicon_bias, metaphor_density, assertiveness, structure_preference, notes }` — only in Voice Profile pipeline.

## 5. Diagnosis

### Why the text feels sterile ("stereo instructions")

1. **No archetype voice injection.** The engine and E.V.E. use static prompts. ArchetypeContract.voice (rhythm, lexicon_bias, metaphor_density, notes) is never passed into the report or Beauty text pipeline. Output is generic "scientific-mythic" without per-archetype tone.

2. **E.V.E. output shape mismatch.** The route forces `{ image, report }`, so E.V.E. cannot return structured ThreeVoice sections. The full EVE_FILTER_SPEC (extract light_signature, archetype, deviations, corrective_vector; rewrite into 3-voice; generate imagery_prompts) is never honored. archetype, deviations, corrective_vector, imagery_prompts are always empty.

3. **Single-string "report" from E.V.E.** The model is asked to produce one string; it likely condenses or summarizes. That string is used as both `fullReport` and `light_signature.raw_signal`, so the user sees a flattened, possibly repetitive output.

4. **Tone constraints are broad.** ENGINE_SPEC says "No metaphors", "No narrative storytelling", "No filler" — which can push toward flat, instructional phrasing. "Cosmology Marbling" adds lists of systems but doesn't specify per-archetype voice.

5. **emotionalSnippet is undifferentiated.** It comes from the same engine call as full_report, with no archetype-specific shaping.

### Smallest leverage point (minimal change)

**File:** `app/api/engine/generate/route.ts`  
**Location:** The user prompt for the report LLM call (line 115)

**Change:** Inject archetype voice into the prompt. Before the call:

1. Derive primary archetype from birth data (or use a lightweight heuristic, e.g. dominant archetype from first engine pass or a simple mapping).
2. Call `getArchetypeAnchor(archetype)` (or `getArchetypeOrFallback(archetype).voice`).
3. Append to the user prompt a short block, e.g.:
   ```
   Voice anchor for this reading: ${anchor.notes}. Rhythm: ${anchor.rhythm}. Lexicon bias: ${anchor.lexicon_bias.join(", ")}. Apply this voice to the emotional_snippet and to the ORACLE sections especially.
   ```

**Why this first:** One file, one prompt change. No schema changes. No E.V.E. rewrite. Improves emotionalSnippet and ORACLE tone immediately. The engine already produces archetype names in the report; you can parse the first "Dominant: X" and use that for the anchor.

**Next step (larger):** Fix the E.V.E. output shape so it returns the full BeautyProfile JSON per EVE_FILTER_SPEC, and pass archetype anchor into the E.V.E. user prompt. That would populate archetype, deviations, corrective_vector, imagery_prompts and allow per-archetype styling there too.
