# LIGS CANONICAL SPEC
## WHOIS HUMAN REGISTRATION REPORT — MVP STRUCTURE

**STATUS:** CANONICAL  
**PURPOSE:** Define the official field structure, naming, and section order for the free and paid WHOIS report system.  
**RULE:** The report must mirror a real WHOIS record first, then expand into LIGS-specific registry fields.  
**RULE:** This is a registry document, not a personality report, beauty report, dossier, or spiritual reading.

---

## 1. OFFICIAL PRODUCT NAMES

| Scope | Name |
|-------|------|
| **FREE OUTPUT** | WHOIS Human Registry Record |
| **PAID OUTPUT** | WHOIS Human Registration Report |
| **SYSTEM** | LIGS Human Identity Registry |

---

## 2. CORE PRODUCT LOGIC

The user runs a WHOIS query on the `/origin` page.

The system returns:

**FREE**
- WHOIS Human Registry Record
- static archetype artifact images
- short archetype expression
- preview of expanded registry fields
- CTA for the full report

**PAID**
- full WHOIS Human Registration Report
- all free record content expanded
- full registry sections
- generated Vector Zero
- generated personal field background
- generated official identity artifact

---

## 3. DOCUMENT PRINCIPLE

The WHOIS Human Registration Report must feel like:
- a formal registry document
- a machine-returned record
- a technical identity registration output
- a WHOIS mirror first
- a LIGS expansion second

It must NOT feel like:
- a beauty report
- a dossier
- a mystical reading
- a self-help report
- a brand brochure

---

## 4. CANONICAL SECTION ORDER

**Paid report** — this section order:

1. WHOIS Header  
2. Registry Record  
3. Registry Classification  
4. Identity Architecture  
5. Solar / Environmental Mapping  
6. Archetype Expression  
7. Registry Interpretation Notes  
8. Registry Artifacts  

**Free report** — this reduced order:

1. WHOIS Header

3. Identity Registration Confirmation

---

## 5. WHOIS HEADER

The document begins with a WHOIS-style header.

**Required fields:**
- WHOIS HUMAN REGISTRATION REPORT *(paid)* / WHOIS HUMAN REGISTRY RECORD *(free)*
- Query: [Subject Name]
- Registry: LIGS Human Identity Registry

**Free version title:** WHOIS HUMAN REGISTRY RECORD  
**Paid version title:** WHOIS HUMAN REGISTRATION REPORT  

---

## 6. REGISTRY RECORD

Official WHOIS mirror block; should visually resemble real WHOIS output.

**Required fields:**
- Registry Record ID  
- Registry Status  
- Created Date  
- Last Updated  
- Subject Name  
- Birth Date  
- Birth Time  
- Birth Location  
- Record Authority  
- Solar Signature  
- Archetype Classification  
- Registry Node  
- Environmental Class  
- Signal Status  

**Example formatting:**

```
WHOIS HUMAN REGISTRATION REPORT
Query: Jane Doe
Registry: LIGS Human Identity Registry

REGISTRY RECORD

Registry Record ID: LIGS-2026-01A93F
Registry Status: Registered
Created Date: 2026-03-12
Last Updated: 2026-03-12

Subject Name: Jane Doe
Birth Date: 1988-03-03
Birth Time: 03:33
Birth Location: Santa Fe, NM

Solar Signature: Fluxionis
Archetype Classification: Flux-Class
Registry Node: LIGS-126
Environmental Class: Transitional Dynamic
Signal Status: Active
Record Authority: LIGS Human Identity Registry
```

---

## 7. REGISTRY CLASSIFICATION

**Purpose:** Formal classification summary of the registered identity.

**Required fields:** Solar Segment, Archetype Class, Environmental Anchor, Behavior Class, Primary Registry Markers  

**Tone:** Concise, technical, categorical.

---

## 8. IDENTITY ARCHITECTURE

**Purpose:** Structural organization of the identity.

**Required fields:** Structural Orientation, Primary Pattern, Load-Bearing Traits, Adaptive Tendencies  

**Tone:** Registry interpretation, not self-help language.

---

## 9. SOLAR / ENVIRONMENTAL MAPPING

**Purpose:** Birth environment and field conditions.

**Required fields:** Solar Environment, Light Conditions, Seasonal Segment, Environmental Pressure, Field Orientation  

**Tone:** Observational, technical, environmental.

---

## 10. ARCHETYPE EXPRESSION

**Purpose:** Human-readable explanation of how the archetype expresses.

**Required fields:** Archetype Classification, Observed Expression, System Role, Common Strengths, Common Friction Points  

**Tone:** Human-readable but registry-consistent. No prompt-engineering vocabulary; no second-person hype.

**Allowed style:**  
*“Fluxionis identities tend to operate in environments where conditions change rapidly…”*

**Disallowed style:**  
*“You are a powerful adaptive being destined to transform everything.”*

---

## 11. REGISTRY INTERPRETATION NOTES

**Purpose:** Practical registry interpretation.

**Required fields:** Integration Notes, Coherence Risks, Stabilization Notes, Developmental Signals  

**Tone:** Operational, applied, useful.

---

## 12. REGISTRY ARTIFACTS

**Purpose:** Official visual attachments to the registration record.

**FREE**
- Archetype Identity Mark (static)  
- Archetype Field Visualization (static)  

**PAID**
- Vector Zero Identity Glyph (generated per user)  
- Personal Field Visualization (generated per user)  
- Official Identity Artifact / Share Card (generated per user)  

Registry Artifacts = official document attachments, not decorative extras.

---

## 13. FREE REPORT RULES

The free WHOIS Human Registry Record must contain:
- WHOIS header  
- Registry Record  
- Identity Registration Confirmation  
- Static Registry Artifacts  
- Short Archetype Expression  
- Registry Extract — Expanded Report Fields  
- CTA  

Must feel complete enough to be real — not a crippled teaser.

---

## 14. PAID REPORT RULES

The WHOIS Human Registration Report must:
- include the full free record structure  
- expand every major registry section  
- include generated artifacts  
- preserve WHOIS registry tone top to bottom  

Paid = authorized, expanded registry record.

---

## 15. CTA LANGUAGE RULE

**Preferred:** Retrieve Full Registration Report; Official WHOIS Human Registration Report; Authorized Registry Report Access; Additional registry fields available in the complete report  

**Avoid:** Buy your report; Unlock your beauty profile; View dossier; Discover your cosmic self  

---

## 16. BANNED LEGACY TERMS

Must not appear in front-facing WHOIS flow or report copy:
- beauty, beauty report, beauty profile  
- dossier  
- reading  
- personality report  

Internal code may retain legacy names temporarily; user-facing copy must not.

---

## 17. FORMATTING RULES

- left-aligned  
- monospace-friendly  
- structured, sparse, official-looking  
- prioritize field labels, registry structure, section headers, machine-returned clarity  
- do not over-style, center, or make it look like a landing page  

---
